## 2. 对组长四项意见的处理

### 2.1 P0：暂存失败可能遗留空目录

已修复 `DocumentProcessingService` 的清理条件。

原逻辑依赖 `stagedFilePath != null`，如果 `stage()` 已创建任务目录、但在 `statObject`、`getObject` 或文件写入过程中失败，变量不会成功赋值，`finally` 将跳过清理。

现在改为：只要任务属于 `PROCESS/REPROCESS`，`finally` 就按 `taskId` 执行幂等清理。

```java
if (heavy) {
    taskFileStagingService.cleanup(msg.taskId);
}
```

因此，即使 `stage()` 中途抛出异常，任务暂存目录也会进入清理流程。

## 3. 文档解析与脏数据清洗升级

解析支持范围保持为：

- PDF：优先 PyMuPDF 文本层，文本不足时按页进入 PaddleOCR。
- DOCX：提取段落、标题层级和表格。
- PPTX：按幻灯片提取文本。
- XLSX：按工作表和行提取表格文本。
- TXT/MD：兼容 UTF-8、GBK、GB18030 等编码。
- PNG/JPG/JPEG/BMP/WEBP：通过 PaddleOCR 解析。

新增的脏文档处理包括：

- 删除零宽字符、控制字符、BOM 和解码替代符。
- 统一全角空格、不换行空格和异常换行。
- 修复 OCR 产生的汉字间异常空格。
- 删除连续重复正文行，但不破坏表格制表符。
- 修复 PDF 句中软换行及英文跨行连字符。
- 识别并删除纯页码。
- 至少 3 页且达到多数页面重复时，才删除顶部或底部的重复页眉页脚，降低误删风险。

## 4. 多策略自适应切片升级

当前切片不再是简单固定字符截断，处理流程为：

```text
结构化 blocks
→ 同页/同章节/同类型短段落合并
→ 根据正文、OCR、表格、列表选择策略
→ 段落/换行/句号/标点/空格递归切分
→ 字符上限与轻量 Token 预算双重保护
→ 安全上下文重叠
→ 低质量短碎片重整
→ 输出带完整元数据的 chunks
```

具体能力：

- 不跨页、章节或内容类型错误合并。
- 表格按行切分，后续切片重复表头。
- 列表优先保持编号条目完整，只有单条过长时才递归拆分。
- OCR、表格、列表使用不同的自适应目标长度。
- 所有切片严格不超过调用方提供的 `chunkSize`。
- 重叠内容同样受字符和 Token 预算限制。
- 内部质量评分只用于安全合并短碎片，不会丢弃原文。
- 保留 `content`、`chunkIndex`、`pageNo`、`chapterPath`、`chunkType` 和 `charCount`。

说明：Token 数为无外部模型依赖的保守估算，不等同于特定大模型 tokenizer 的精确结果。

## 7. F6 配置责任与最终来源

| 配置项 | 当前来源 | 当前责任边界 | 是否动态生效 |
|---|---|---|---|
| 文件大小上限 | Worker `TaskFileStagingService` 固定 50MB；FastAPI `MAX_FILE_SIZE_MB` 默认 50MB | F4/Worker 与部署环境共同保证；尚未统一由 F6 下发 | 否 |
| PDF 页数上限 | `taskPayloadJson.maxPdfPages` 与 FastAPI `MAX_PDF_PAGES` 默认 200，取较小值 | 知识库任务快照 + F4 部署保护 | 随任务生效，但不是 F6 系统配置 |
| OCR 开关 | `taskPayloadJson.enableOcr` | 知识库策略/任务快照决定，F4 执行 | 随任务生效 |
| OCR CPU 线程 | FastAPI 环境变量 `OCR_CPU_THREADS`，默认 2 | 部署环境负责 | 否，需重启容器 |
| FastAPI 连接超时 | Worker `AI_CONNECT_TIMEOUT_MS`，默认 5000ms | Worker/部署环境负责 | 否，需重启 Worker |
| FastAPI 读取超时 | Worker `AI_READ_TIMEOUT_MS`，默认 30000ms | Worker/部署环境负责 | 否，需重启 Worker |
| 最大解析并发 | F6 `parser.max_concurrent_tasks` → Admin 接口/配置事件 → `DynamicConfigHolder` | F6 负责配置，Worker 负责执行 | **是** |
| F6 `timeoutSeconds` | Worker 启动时可以读取并记录，但 `FastApiClient` 仍使用环境变量超时 | F6 与 Worker 后续联调项 | **尚未接入执行** |
| F6 `maxRetryCount` | Worker 启动时可以读取并记录，但当前没有写入 `DynamicConfigHolder` 或重试判断 | F6 与 Worker 后续联调项 | **尚未接入执行** |

因此当前不能宣称“文件限制、OCR 限制和超时均已由 F6 动态控制”。准确结论是：

> F6 动态并发配置已经生效；其余资源保护主要由任务快照和环境变量控制，`timeoutSeconds`、`maxRetryCount` 尚需 Worker/F6 后续接线。
