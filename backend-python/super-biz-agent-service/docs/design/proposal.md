# 智能问答系统未实现功能需求提案

## 1. 文档目的

本文基于《智能问答需求分析(2).docx》与当前项目实现情况进行差距分析，整理当前项目尚未实现或仅部分实现的产品化功能，作为后续迭代开发需求依据。

当前项目已具备基础智能对话、SSE 流式输出、前端本地多会话历史、Markdown/TXT 文件上传入库、Milvus 向量检索、Trace、Bad Case 等能力。但原需求文档面向的是带登录权限、管理后台、可配置 RAG、引用溯源和运营统计的智能问答系统，当前项目在产品化管理能力上仍有明显缺口。当前版本采用单一登录用户权限模型，所有已登录用户都可以使用管理后台能力。

## 2. 当前已部分具备的能力

- 基础问答接口：已提供 `/api/chat` 与 `/api/chat_stream`。
- 流式输出：已支持 SSE 流式返回内容。
- 会话隔离：后端基于 `session_id/thread_id` 保存会话上下文，前端会生成独立会话 ID。
- 前端会话历史：已支持新建对话、切换历史对话、删除历史对话，本地使用 `localStorage` 持久化。
- 会话标题：前端已基于首条用户消息截断生成标题。
- 文件上传入库：已支持 TXT、Markdown 文件上传并自动向量化索引。
- RAG 基础检索：已基于 Milvus 与本地知识检索工具实现 Top K 向量召回。
- 多轮上下文：已基于 LangGraph MemorySaver 维护会话上下文。

## 3. 未实现功能需求

### 3.1 身份认证与登录模块

#### 3.1.1 账号体系

当前项目未实现用户注册、登录、密码校验和用户表。

需求：

- 支持用户注册。
- 支持用户名 + 密码登录。
- 密码需加盐哈希存储，不允许明文保存。
- 登录接口返回可用于后续访问的认证凭证，例如 Session Cookie 或 JWT。
- 支持退出登录。

验收标准：

- 未注册用户不能直接登录。
- 密码错误时返回明确但不泄露安全细节的错误提示。
- 登录成功后可访问授权业务接口。

#### 3.1.2 全接口强制鉴权

当前项目所有业务接口基本处于免登录访问状态。

需求：

- 对 `/api/chat`、`/api/chat_stream`、`/api/upload`、`/api/index_directory`、`/api/badcases` 等业务接口统一增加鉴权中间件或依赖。
- 未登录用户禁止访问所有业务功能。
- 静态首页可访问，但进入业务页面前应校验登录态。

验收标准：

- 未登录请求业务接口返回 401。
- 已登录请求携带有效凭证时正常访问。
- 过期或伪造凭证不能访问业务接口。

#### 3.1.3 会话持久化与登录态区分

当前项目仅有问答会话 ID，不存在用户登录会话。

需求：

- 登录后保持会话，页面刷新无需重复登录。
- 支持登录态过期策略。
- 当前版本采用单一登录用户权限模型，不设置额外管理角色。
- 后端所有业务数据需绑定用户身份，避免用户之间读取或删除彼此会话。

验收标准：

- 页面刷新后仍能识别当前登录用户。
- 已登录用户可访问管理后台接口。
- 不同用户无法通过猜测 `session_id` 获取其他用户的会话历史。

#### 3.1.4 登录态权限控制

当前项目未实现登录态权限控制。

需求：

- 未登录用户不能访问业务接口和管理接口。
- 已登录用户可使用智能问答、查看自己的对话历史。
- 已登录用户也可使用后台配置、知识库管理、统计看板等管理功能。
- 权限校验应在后端强制执行，前端入口控制只作为体验优化。

验收标准：

- 未登录用户访问后台接口返回 401。
- 已登录用户可进入后台管理页。
- 后台管理操作必须携带有效登录凭证。

### 3.2 用户端智能对话增强

#### 3.2.1 服务端多会话管理

当前前端已用 `localStorage` 保存会话列表，但缺少服务端持久化会话列表和用户绑定。

需求：

- 服务端保存用户的会话列表、会话标题、创建时间、更新时间、消息明细。
- 登录后显示当前用户全部历史会话。
- 历史会话按最新消息时间倒序排列。
- 删除会话时同步删除服务端记录或标记为已删除。
- 不同会话上下文完全隔离，且不同用户之间数据隔离。

验收标准：

- 换浏览器或清空本地缓存后，登录仍能看到历史会话。
- 删除会话后刷新页面不再出现。
- 用户 A 无法访问用户 B 的会话。

#### 3.2.2 输入体验增强

当前前端使用单行 `input`，虽然监听了 `Shift+Enter`，但单行输入框不支持多行编辑体验。

需求：

- 将输入框改为多行输入组件。
- 支持 `Shift+Enter` 换行。
- 支持 `Enter` 发送消息。
- 保持移动端和桌面端输入体验一致。

验收标准：

- 输入多行问题时不会误发送。
- 按 Enter 可发送消息。

#### 3.2.3 生成中断与重新生成

当前项目没有前端中断控制器，也没有重新生成入口。

需求：

- 回答生成过程中支持用户主动停止生成。
- 停止生成后保留用户问题和已生成内容。
- 模型异常、网络超时或服务错误时，界面展示友好错误提示。
- 错误时保留用户已提交问题，并提供“重新生成”入口。
- 重新生成应复用同一用户问题和同一会话上下文。

验收标准：

- 流式生成时点击停止可立即终止前端读取。
- 服务端异常时前端不会丢失用户输入。
- 点击重新生成可重新请求回答。

#### 3.2.4 上下文自动截断与压缩策略产品化

当前项目存在保留最近若干消息的修剪函数，但没有真正接入到 Agent 执行流程，也没有按模型上下文窗口进行动态截断或摘要压缩。

需求：

- 按模型上下文窗口动态计算可保留历史消息。
- 优先保留最新问答轮次。
- 对较早历史进行摘要压缩，而不是固定保留 3 轮。
- 将压缩摘要写入会话上下文，保证连续追问可用。
- 支持配置最大上下文轮次、摘要触发阈值和摘要模型。

验收标准：

- 长会话连续追问时仍能保留核心背景。
- 上下文过长不会导致模型调用失败。
- 会话压缩过程可观测、可记录。

### 3.3 意图识别与路由

当前项目依赖 Agent 自主选择工具，没有显式的业务意图识别和路由链路。

需求：

- 增加意图识别模块，识别知识问答、通用闲聊、普通对话等类型。
- 知识问答意图进入 RAG 检索流程。
- 通用闲聊或普通对话直接调用大模型，不触发知识库检索。
- 意图识别结果应返回给前端，用于思考过程展示。
- 支持低置信度兜底策略。

验收标准：

- 闲聊问题不会触发知识库检索。
- 电力规范、技术标准类问题会进入 RAG 流程。
- 返回数据包含本轮识别的意图类型和置信度。

### 3.4 RAG 检索能力增强

#### 3.4.1 检索模式配置

当前项目仅支持基础向量检索，未实现重排序模式。

需求：

- 支持纯语义向量检索。
- 支持向量检索 + 重排序双模式。
- 已登录用户可配置当前系统使用的检索模式。
- 重排序模型、重排序接口和超时时间可配置。

验收标准：

- 关闭重排序时仅返回向量召回结果。
- 开启重排序时结果按重排序分数重新排序。
- 重排序异常时可降级到向量检索。

#### 3.4.2 多知识库联合检索

当前项目只有统一的 `biz` collection，未实现多知识库选择和并行检索。

需求：

- 支持创建多个知识库，例如术语库、技术监督业务库、测试库等。
- 用户可选择一个或多个知识库参与检索。
- 多库并行检索后合并、去重、排序。
- 检索结果需保留所属知识库信息。

验收标准：

- 可对单个知识库检索。
- 可同时选择多个知识库并合并结果。
- 返回引用来源中能看到知识库名称。

#### 3.4.3 阈值与条数配置

当前项目只通过环境配置固定 `rag_top_k`，没有相似度阈值、重排序阈值和动态参数配置。

需求：

- 支持配置向量召回 Top K。
- 支持配置相似度阈值，低于阈值的片段过滤。
- 支持配置重排序后保留最大条数。
- 支持配置重排序分数阈值。
- 参数修改后实时生效，无需重启服务。

验收标准：

- 修改 Top K 后下一次检索立即生效。
- 低于阈值的召回片段不会进入答案生成。
- 参数配置有默认值和合法范围校验。

#### 3.4.4 无资料策略

当前项目检索不到资料时仅返回工具文本“没有找到相关信息”，没有面向用户的统一无资料策略。

需求：

- 当无有效召回切片时，优先提示“当前知识库未检索到足够依据”。
- 支持已登录用户配置是否允许兜底通用回答。
- 若开启兜底回答，回答必须明确标记“非知识库依据”。
- 无资料事件需记录到问答日志，便于后续知识库补充。

验收标准：

- 无召回时不会伪造知识库依据。
- 开启兜底时，前端可明确看到非知识库依据标记。

### 3.5 思考过程可视化

当前聊天接口只返回内容、工具调用和错误等基础事件，未按原需求展示完整业务链路节点。

需求：

- 知识问答链路展示：意图识别、术语匹配、知识库召回、答案生成。
- 普通对话链路展示：意图识别、调用大模型、答案生成。
- 每个节点展示状态：等待中、处理中、成功、失败、跳过。
- 支持思考过程面板展开和折叠。
- 流式生成完成后默认自动折叠。

验收标准：

- 前端能实时展示每个节点状态变化。
- 普通对话和知识问答展示不同处理链路。
- 完成后面板自动折叠，不遮挡主对话。

### 3.6 引用溯源

当前知识检索工具会把来源文本塞给模型，但最终回答没有结构化引用、上角标、引用详情弹窗和原文下载。

需求：

- 回答正文中使用 `[1][2]` 形式标注引用来源。
- 引用序号与候选切片召回排序一致。
- 每个引用包含来源文档、所属章节、切片内容、相似度分数、重排序分数、知识库名称。
- 点击引用序号可查看来源详情。
- 支持下载原文文件。
- 同一文档同一切片连续多处引用时合并标注，避免重复编号。

验收标准：

- RAG 回答必须返回 `answer` 与 `citations` 两类结构化数据。
- 前端可点击引用编号查看详情。
- 引用详情中的切片内容与检索结果一致。
- 可下载或访问对应原文。

### 3.7 管理后台

当前项目没有独立管理后台页面和后台接口。

需求：

- 新增后台管理入口，登录后可见。
- 后台应至少包含系统设置、模型配置、RAG 参数配置、知识库管理、问答运营统计等模块。
- 后台接口必须强制登录校验。

验收标准：

- 未登录用户访问后台接口返回 401。
- 已登录用户可进入后台页面。

### 3.8 系统参数配置

当前配置主要来自 `.env` 和 `Settings`，修改后通常需要重启服务。

需求：

- 提供系统参数配置页。
- 参数修改保存后实时生效，无需重启服务。
- 支持一键重置为系统默认参数。
- 每项参数附带说明提示、默认值、合法范围和是否敏感。
- 敏感参数如 API Key 前端脱敏展示。

验收标准：

- 修改参数后下一次请求生效。
- 重置默认值后系统恢复默认配置。
- 未登录用户无法读取或修改系统参数。

### 3.9 大模型配置

当前项目固定使用 DashScope 相关配置，未提供后台动态配置和多模型兼容能力。

需求：

- 已登录用户可配置 LLM 接口地址。
- 已登录用户可配置 API Key、模型名称、接口超时时间。
- 支持主流开源或商用大模型接入。
- 支持配置测试连通性。
- 支持区分对话模型、摘要模型、重排序模型等用途。

验收标准：

- 修改模型配置后无需重启即可生效。
- 连通性测试能返回成功或失败原因。
- API Key 不在前端明文展示。

### 3.10 知识库管理

当前项目只有文件上传入口和目录索引接口，缺少知识库维度管理。

需求：

- 支持创建、编辑、删除知识库。
- 支持上传文档到指定知识库。
- 支持查看文档列表、切片数量、索引状态、上传人、更新时间。
- 支持重新索引、删除文档、下载原文。
- 支持知识库启用和停用。
- 支持术语库、技术监督业务库、测试库等多库绑定。

验收标准：

- 上传文档时必须选择目标知识库。
- 删除文档后对应切片不可再被检索。
- 停用知识库后不参与用户问答检索。

### 3.11 总览与问答运营统计

当前项目未实现运营统计仪表盘。

需求：

- 管理总览仪表盘展示核心指标卡片。
- 指标包括用户总数、知识库数量、文档总量、切片总量、知识问答总次数。
- 提供近 30 天知识问答日活趋势图。
- 支持按日期范围筛选统计数据。
- 问答日志需记录用户、会话、问题、意图、是否命中知识库、引用数量、耗时、状态。

验收标准：

- 已登录用户可查看总览页面。
- 指标数据与数据库记录一致。
- 趋势图能展示近 30 天问答使用量变化。

## 4. 建议优先级

### P0：产品化访问控制与数据隔离

- 用户注册、登录、退出。
- 全接口鉴权。
- 单一登录用户权限模型。
- 服务端会话持久化与用户数据隔离。

原因：这是后续后台、知识库和统计能力的安全底座。

### P1：核心问答体验补齐

- 显式意图识别与路由。
- RAG 阈值过滤与无资料策略。
- 引用溯源结构化返回和前端展示。
- 生成中断、重新生成、友好错误恢复。
- 多行输入体验。

原因：这些功能直接影响用户对智能问答可信度和可用性的感知。

### P2：RAG 管理能力

- 多知识库管理。
- 多库联合检索。
- 重排序模式。
- RAG 参数后台配置并实时生效。
- 文档列表、切片状态、重新索引和原文下载。

原因：当前 RAG 已有基础能力，补齐管理层后才能支撑真实业务知识库运营。

### P3：管理后台与运营统计

- 系统参数配置。
- 大模型配置。
- 管理总览指标卡片。
- 近 30 天问答趋势图。
- 问答日志和统计分析。

原因：这些能力偏运营和管理，可在认证、问答和知识库主链路稳定后建设。

## 5. 推荐数据模型方向

后续实现可考虑新增以下核心表或集合：

- `users`：用户账号、密码哈希、状态、创建时间。
- `sessions`：登录会话或 Token 黑名单。
- `chat_sessions`：问答会话标题、所属用户、更新时间、删除状态。
- `chat_messages`：用户消息、助手消息、引用信息、耗时、状态。
- `knowledge_bases`：知识库名称、描述、状态、创建人。
- `knowledge_documents`：文档元信息、所属知识库、索引状态、切片数量、原文路径。
- `knowledge_chunks`：切片元信息、来源章节、向量库 ID、分数信息。
- `system_settings`：动态系统配置。
- `qa_metrics`：问答统计日志。

## 6. 非目标说明

以下能力虽然当前项目已有或与原需求不完全一致，本提案不作为缺口重点：

- Trace、Bad Case：这些属于当前项目额外增强能力，不在原智能问答需求文档的核心缺口内。
- MCP 工具治理：当前项目已具备基础工具治理能力，不作为本次“未实现功能”主线。
- 评测集和演示脚本：当前项目已有基础文档和评测样例，后续可随功能迭代补充。
